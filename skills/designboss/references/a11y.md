# Accessibility Audit — Design Boss Reference

## When to Load
Load this reference when:
- The user asks about accessibility (`/designboss a11y` or `/designboss accessibility`)
- You notice potential accessibility issues during any design review
- The project serves a broad audience or has compliance requirements
- Evaluating forms, navigation, or interactive elements

## Accessibility Evaluation Framework

### Visual Accessibility
- **Color contrast:** WCAG 2.1 AA minimum (4.5:1 for text, 3:1 for large text/UI elements)
- **Don't rely on color alone** to convey meaning — use shape, text, icons too
- **Text sizing:** Supports Dynamic Type / user font size preferences
- **Motion:** Respect `prefers-reduced-motion` — provide static alternatives
- **Dark mode:** Accessible contrast ratios in both light and dark themes

### Screen Reader / VoiceOver
- All interactive elements have meaningful labels (not "Button 1")
- Reading order makes logical sense (not just visual order)
- Images have alt text (decorative images are hidden from assistive tech)
- State changes are announced (loading, errors, success)
- Custom components expose proper accessibility roles and traits

### Keyboard & Alternative Input
- All functionality reachable via keyboard alone
- Visible focus indicators (not just browser defaults that get removed)
- Logical tab order follows visual flow
- No keyboard traps (can always tab out of any component)
- Touch targets minimum 44×44pt (Apple) / 48×48dp (Android)

### Cognitive Accessibility
- Clear, simple language in labels and instructions
- Error messages explain what went wrong AND how to fix it
- Consistent navigation patterns throughout
- No time limits without user control
- Complex interactions have simpler alternatives

### Platform-Specific (Apple)
- VoiceOver rotor actions for custom interactions
- Accessibility traits properly set (.isButton, .isHeader, etc.)
- AX audit in Xcode shows no warnings
- Switch Control and Voice Control compatible
- Haptic feedback supplements visual feedback

### Spatial Accessibility (visionOS)
- Pointer accessibility for users who can't use eye tracking
- Voice Control for hands-free interaction
- Switch Control support for motor impairments
- Dwell control as alternative to pinch gestures
- Head pointer as alternative to eye tracking

## Automated vs Manual Testing

**"Automated tools catch roughly 30% of accessibility issues — you catch the other 70%."**

- **Automated catches:** Contrast violations, missing alt text, missing form labels, missing landmarks
- **Manual only:** Focus order, reading order, ARIA misuse, cognitive barriers, custom component keyboard patterns, dynamic content announcements, screen reader experience quality

## Screen Reader Testing Protocol

```
Setup:
- Screen Reader: [VoiceOver / NVDA / JAWS]
- Browser: [Safari / Chrome / Firefox]
- OS: [macOS / Windows / iOS / Android]

Navigation Testing:
- [ ] Heading structure is logical and hierarchical (h1 -> h2 -> h3)
- [ ] Landmark regions present and labeled (main, nav, banner, contentinfo)
- [ ] Skip links allow jumping to main content
- [ ] Tab order moves in logical sequence
- [ ] Focus indicator always visible and clear

Interactive Component Testing:
- [ ] Buttons announced with role and label; state changes announced
- [ ] Links distinguishable from buttons; destination clear from label
- [ ] Forms: labels associated, required fields announced, errors identified
- [ ] Modals: focus trapped, Escape closes, focus returns on close
- [ ] Custom widgets (tabs, accordions, menus): proper ARIA roles and keyboard patterns

Dynamic Content Testing:
- [ ] Status messages announced via live regions without focus change
- [ ] Loading states communicated to screen reader users
- [ ] Error messages announced immediately and associated with field
- [ ] Toast/notifications announced via aria-live and dismissible
```

## Keyboard Navigation Checklist

```
Global:
- [ ] All interactive elements reachable via Tab
- [ ] Tab order follows visual layout logic
- [ ] Skip navigation link present and functional
- [ ] No keyboard traps (can always Tab away)
- [ ] Focus indicator visible on every interactive element
- [ ] Escape closes modals, dropdowns, and overlays
- [ ] Focus returns to trigger element after modal/overlay closes

Tabs:
- [ ] Tab key moves focus into/out of tablist and into active tabpanel
- [ ] Arrow keys move between tab buttons
- [ ] Home/End move to first/last tab
- [ ] Selected tab indicated via aria-selected

Menus:
- [ ] Arrow keys navigate menu items
- [ ] Enter/Space activates menu item
- [ ] Escape closes menu and returns focus to trigger

Data Tables:
- [ ] Headers associated with cells via scope or headers attributes
- [ ] Caption or aria-label describes table purpose
- [ ] Sortable columns operable via keyboard
```

## Common ARIA Anti-Patterns
- `aria-label` on non-interactive elements
- Redundant roles on semantic HTML (`role="button"` on `<button>`)
- `aria-hidden="true"` on focusable elements
- React portals breaking focus order
- SPA route changes not announcing page titles

## Severity Classification
- **Critical**: Blocks access entirely for some users (e.g., no keyboard access to primary action)
- **Serious**: Major barriers requiring workarounds (e.g., missing form labels)
- **Moderate**: Causes difficulty but has workarounds (e.g., low contrast on secondary text)
- **Minor**: Annoyances that reduce usability (e.g., inconsistent focus styles)
